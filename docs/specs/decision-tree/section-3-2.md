## ⚠ DESIGN DECISION: GOAL GEOMETRY SOURCE

**Problem identified during §3.2 drafting:**
`GoalOpeningScore` (required for SHOOT utility) depends on the opponent goal position.
Neither `PerceptionSnapshot` nor `MatchContext` contains goal position data.

**Decision:** Goal positions are static pitch geometry — they are fixed world constants,
not perceptual data. An agent always knows where the goals are. They do not require
perception filtering and must not be stored as agent-perceived data.

**Resolution:** A `PitchGeometry` static class is defined in this section (§3.2.1.4),
owned by Decision Tree Specification #8. It exposes fixed pitch constants including
goal centre positions and post positions. The DT accesses these directly. This is
not a violation of the no-omniscience constraint (KD-5, DT Spec §1.4) — goals are
fixed landmarks equivalent to pitch markings.

`MatchContext` is not modified. No change to `PerceptionSnapshot` is required. No
cross-spec amendment to Perception System #7 is triggered. This is a contained
clarification within the DT specification's own scope.

## 3.2.0 Scoring in the Pipeline

Option scoring is **Step 4** of the 6-step Decision Tree pipeline. It receives the complete
`ActionOption[]` list produced by `GenerateOptions()` (Step 3 / §3.1) and assigns a
`ScoredUtility` value in [0.01, 1.0] to each option. The scored list is then consumed
by `SelectAction()` (Step 5 / §3.3).

```
Pipeline context:

  Step 3: GenerateOptions()    → ActionOption[] (ScoredUtility = 0.0 for all; BaseUtility = 0.0)
  Step 4: ScoreOptions()       → ActionOption[] (ScoredUtility populated; BaseUtility unchanged)
  Step 5: SelectAction()       → consumes ScoredUtility; applies composure noise; picks winner
```

`ScoreOptions()` iterates the option list once. For each option, it dispatches to the
appropriate formula function based on `option.ActionType`. Each formula reads from
`DecisionContext` (which contains a snapshot copy, MatchContext, TacticalContext, and
AgentState) — it may not access any external state. This is enforced by the static
analysis constraint (DT Spec §1.6.1: `DecisionTree.cs` may not import `WorldState`
namespace).

**No option is modified in any field other than `ScoredUtility`.** All other option fields
(populated in §3.1) are read-only inputs to the scoring formulas.

## 3.2.1 Universal Scoring Foundations

### 3.2.1.1 Master Utility Formula Structure

All seven utility formulas share the same four-component multiplicative structure:

```
U(option) = BaseUtility(option)
          × AttributeMultiplier(option)
          × ContextModifier(option)
          × (1 − RiskPenalty(option))

clamped to [0.01, 1.0]
```

**Component definitions:**

| Component | Definition | Range | Notes |
|-----------|-----------|-------|-------|
| `BaseUtility` | Intrinsic value of this action type in the current game state. Varies with `FieldZone` and `PossessionState` only. | [0.05, 0.90] | Zone modifier table in §3.2.1.3. All [GT]. |
| `AttributeMultiplier` | Product of attribute exponent terms. Scales how well the agent can *execute* this action type, independent of game state. | [0.36, 1.73] | See §3.2.1.2 for derivation bounds. |
| `ContextModifier` | Additional scaling from match context: tactical instructions, threat-specific scores, lane quality, space quality, feasibility. | [0.0, 1.0] for quality scores; [0.6, 1.4] for tactical multipliers | Action-specific; each defined in its formula section. |
| `1 − RiskPenalty` | Reduces utility when action failure is likely. RiskPenalty ∈ [0.0, 0.4] across all formulas — never eliminates an option entirely. | [0.60, 1.00] | All [GT] penalty coefficients. |

**Multiplication order matters for numerical precision:** evaluate in the order shown
above. All intermediate results are float32; no intermediate clamping is applied.
The final clamp to [0.01, 1.0] occurs once, after all four components are multiplied.

**Why multiplicative, not additive?** A multiplicative structure ensures that a very
low attribute score does not merely *reduce* an option's appeal — it multiplies it.
An agent with `Technique = 1` executing a long pass under pressure receives attribute,
context, and risk penalties simultaneously, producing a very low final score without
any single component needing to reach zero. Additive weighting would require careful
normalisation to prevent one component from dominating; the multiplicative form is
self-normalising.

### 3.2.1.2 Attribute Normalisation

All attributes are normalised from their raw [1, 20] integer value to a float in (0, 1]
before use in any formula. The normalisation is defined once here and applies identically
across all seven formulas.

**Standard normalisation formula:**

```
A_x = (x_raw − 1) / 19

where:
  x_raw ∈ {1, 2, 3, ..., 20}     (integer PlayerAttribute value)
  A_x   ∈ {0.0, 0.0526, ..., 1.0} (normalised float)
```

**Key values:**

| x_raw | A_x | Notes |
|-------|-----|-------|
| 1 | 0.0000 | Absolute floor — no contribution from this attribute |
| 5 | 0.2105 | Below-average player |
| 10 | 0.4737 | Average player (≈ 0.5) |
| 15 | 0.7368 | Above-average player |
| 20 | 1.0000 | Elite ceiling |

**Normalisation is applied once per formula invocation.** If the same attribute is
used in multiple formulas (e.g., `Composure` appears in SHOOT and HOLD), the
normalised value must be computed from the same `PlayerAttributes` struct field
each time — no caching of normalised values across formulas.

**Exponent application:** Each formula applies a power exponent to A_x before use.
The general term is `A_x ^ exp` where `exp` is a [GT] constant specific to each
attribute's influence role. The exponent governs **sensitivity** — the steepness of
the attribute response curve:

- `exp = 1.0`: linear response (doubling A_x doubles the multiplier contribution)
- `exp < 1.0`: diminishing returns (low attributes are penalised more than proportionally;
  used for attributes where minimal competence matters more than elite proficiency)
- `exp > 1.0`: accelerating returns (elite attributes provide disproportionate benefit;
  not used in Stage 0 formulas — reserved for Stage 1 differentiation tuning)

**AttributeMultiplier range derivation:**
Each formula defines an `AttributeMultiplier` as a product of terms in the form
`A_x ^ exp` or `(0.5 + A_x × 0.5)` (shifted to prevent zero-floor). The range
at attribute extremes:

| Scenario | Attribute values | AttributeMultiplier |
|----------|-----------------|---------------------|
| All attributes = 1 (A_x = 0.0) | x_raw = 1 for all | Near 0 (or 0.5 if shifted form used) |
| All attributes = 20 (A_x = 1.0) | x_raw = 20 for all | 1.0 (product of 1.0^exp terms = 1.0) |

**Zero-floor problem and the shifted form:**
`A_x ^ exp` at A_x = 0.0 returns 0.0 regardless of exponent. This means an attribute
of 1 in any contributing attribute would drive `AttributeMultiplier` to 0.0, and
consequently `U(option) = 0.0`, violating the FR-07 clamp floor constraint (the clamp
prevents 0.0 from propagating, but the option would floor at 0.01 regardless of how
good the other factors are).

**Policy:** Attributes with low `x_raw = 1` expected in playable squad ranges (Finishing,
Long Shots) use the shifted form: `(0.5 + A_x × 0.5)`, which maps [0,1] → [0.5, 1.0].
Attributes representing genuine near-zero competence (e.g., Dribbling for a goalkeeper
in an outfield role) use the raw `A_x ^ exp` form, accepting the near-zero floor as
correct behaviour — such agents genuinely should not dribble.

**Shifted form values:**

| x_raw | A_x | Shifted: (0.5 + A_x × 0.5) |
|-------|-----|--------------------------|
| 1 | 0.0000 | 0.5000 |
| 5 | 0.2105 | 0.6053 |
| 10 | 0.4737 | 0.7368 |
| 15 | 0.7368 | 0.8684 |
| 20 | 1.0000 | 1.0000 |

The shifted form is identified explicitly in each formula where it is used. All others
use the raw `A_x ^ exp` form.

### 3.2.1.3 Zone Modifier Table

`BaseUtility` varies by the ball's current `FieldZone` (`DecisionContext.BallZone`).
The zone modifier is a multiplier on the nominal `BaseUtility` constant. Agents in
their attacking third should be encouraged to shoot and dribble; agents in their own
third should be discouraged from risky actions and encouraged to clear possession.

The agent's own zone is defined relative to their attacking direction:
- `DEFENSIVE`: 0m – 35m from own goal line
- `MIDFIELD`: 35m – 65m
- `ATTACKING`: 65m – 105m

**Note:** These zones are ball-zone derived (`MatchContext.BallZone`), not agent-zone
derived. The ball's zone determines the tactical context, not the agent's current
position. An agent who is physically in the attacking third while the ball is in midfield
operates under midfield zone modifiers.

**Zone Modifier Table:**

| Action | DEF Zone | MID Zone | ATT Zone | Rationale |
|--------|----------|----------|----------|-----------|
| PASS | 1.05 [GT] | 1.00 | 0.90 [GT] | Passing is most critical and cautious in defence; slightly less urgent in attack where shot/dribble are preferred |
| SHOOT | 0.10 [GT] | 0.55 [GT]¹ | 1.00 | Strongly discourage shots from own half; midfield long shots require Long Shots attribute gate (§3.2.3.1) |
| DRIBBLE | 0.70 [GT] | 1.00 | 1.10 [GT] | Dribbling in defence is risky; in attack it is a valid creative option |
| HOLD | 1.25 [GT] | 1.00 | 0.80 [GT] | Holding is safest in defence; in attack an agent who only holds is wasting chance creation |
| MOVE_TO_POSITION | 1.00 | 1.00 | 1.00 | Positional duty is constant regardless of ball zone — no zone incentive/penalty |
| PRESS | 0.80 [GT] | 1.00 | 1.20 [GT] | Pressing from a defensive position can expose space; pressing high up the pitch is tactically sound |
| INTERCEPT | 1.10 [GT] | 1.00 | 0.90 [GT] | Interception is most valuable defensively; in attack the agent should be creating, not chasing |

¹ SHOOT midfield zone modifier (0.55) applies only when `A_LongShots > LONG_SHOT_THRESHOLD`
(§3.2.3.1). Otherwise, the modifier is 0.05 [GT] — effectively suppressing midfield shots
for players without long-shot ability.

**Application in formulas:**
In each formula, `ZoneModifier` is read from this table using `DecisionContext.BallZone`.
It is applied as a multiplier to the nominal `BaseUtility_nominal` constant:

```
BaseUtility = BaseUtility_nominal × ZoneModifier(BallZone)
```

This is the only table lookup in the scoring pipeline. All other values are computed.

### 3.2.1.4 PitchGeometry Static Class

Goal positions are static pitch constants. They are not perceptual data and are not
subject to the `PerceptionSnapshot` no-omniscience constraint (KD-5). All agents always
know where the goals are — this is equivalent to knowing the pitch dimensions.

```csharp
/// <summary>
/// PitchGeometry — static constants for standard football pitch geometry.
///
/// Coordinate system (consistent with Ball Physics #1 §2.2 and Agent Movement #2 §2.1):
///   Origin (0, 0) = centre of pitch
///   X-axis: pitch length (−52.5m to +52.5m; total 105m)
///   Y-axis: pitch width (−34m to +34m; total 68m)
///
/// Goal positions are defined for HOME team attacking in the +X direction.
/// Away team attacks in the −X direction.
/// The Decision Tree must query the correct goal based on agent team affiliation
/// (DecisionContext.AgentState.TeamId).
///
/// Owned by: Decision Tree Specification #8.
/// No changes to MatchContext or PerceptionSnapshot required.
/// </summary>
public static class PitchGeometry
{
    // ── Pitch Dimensions ─────────────────────────────────────────────────────────

    public const float PITCH_LENGTH_M  = 105.0f;   // [FIXED] Standard UEFA pitch length
    public const float PITCH_WIDTH_M   = 68.0f;    // [FIXED] Standard UEFA pitch width
    public const float HALF_LENGTH_M   = 52.5f;    // [FIXED] Origin to own/opponent goal
    public const float HALF_WIDTH_M    = 34.0f;    // [FIXED] Origin to sideline

    // ── Goal Geometry ─────────────────────────────────────────────────────────────

    public const float GOAL_WIDTH_M    = 7.32f;    // [FIXED] Standard goalpost separation
    public const float GOAL_HALF_WIDTH = 3.66f;    // [FIXED] Half goal width

    // Home team attacks toward +X; their opponent goal is at x = +52.5m
    public static readonly Vector2 HOME_OPPONENT_GOAL_CENTRE = new Vector2( 52.5f, 0.0f);
    public static readonly Vector2 HOME_OPPONENT_GOAL_POST_L = new Vector2( 52.5f, +3.66f);
    public static readonly Vector2 HOME_OPPONENT_GOAL_POST_R = new Vector2( 52.5f, -3.66f);

    public static readonly Vector2 HOME_OWN_GOAL_CENTRE      = new Vector2(-52.5f, 0.0f);
    public static readonly Vector2 HOME_OWN_GOAL_POST_L      = new Vector2(-52.5f, +3.66f);
    public static readonly Vector2 HOME_OWN_GOAL_POST_R      = new Vector2(-52.5f, -3.66f);

    // Away team attacks toward −X; their opponent goal is at x = −52.5m
    public static readonly Vector2 AWAY_OPPONENT_GOAL_CENTRE = new Vector2(-52.5f, 0.0f);
    public static readonly Vector2 AWAY_OPPONENT_GOAL_POST_L = new Vector2(-52.5f, +3.66f);
    public static readonly Vector2 AWAY_OPPONENT_GOAL_POST_R = new Vector2(-52.5f, -3.66f);

    public static readonly Vector2 AWAY_OWN_GOAL_CENTRE      = new Vector2( 52.5f, 0.0f);
    public static readonly Vector2 AWAY_OWN_GOAL_POST_L      = new Vector2( 52.5f, +3.66f);
    public static readonly Vector2 AWAY_OWN_GOAL_POST_R      = new Vector2( 52.5f, -3.66f);

    // ── Helper: Get opponent goal centre for a given team ─────────────────────────
    //
    // Called by §3.2.3 SHOOT utility. TeamId: 0 = home, 1 = away.
    //
    public static Vector2 GetOpponentGoalCentre(int teamId) =>
        teamId == 0 ? HOME_OPPONENT_GOAL_CENTRE : AWAY_OPPONENT_GOAL_CENTRE;

    public static Vector2 GetOpponentGoalPostL(int teamId) =>
        teamId == 0 ? HOME_OPPONENT_GOAL_POST_L : AWAY_OPPONENT_GOAL_POST_L;

    public static Vector2 GetOpponentGoalPostR(int teamId) =>
        teamId == 0 ? HOME_OPPONENT_GOAL_POST_R : AWAY_OPPONENT_GOAL_POST_R;
}
```

**XC-NOTE:** Coordinate system must be verified consistent with Ball Physics #1 §2.2.
If Ball Physics uses a different origin convention (e.g., corner at (0,0) rather than
centre-pitch), these constants must be adjusted. Flagged for §9 Approval Checklist
cross-check (XC-GEOM-01).

### 3.2.1.5 Utility Clamp Rule

All utility formulas are subject to a final clamp:

```
ScoredUtility = Clamp(U_raw, 0.01f, 1.0f)

where U_raw = BaseUtility × AttributeMultiplier × ContextModifier × (1 − RiskPenalty)
```

**Floor (0.01):** Prevents any generated option from having zero utility. A candidate
produced by `GenerateOptions()` has already passed all eligibility gates — it is a
viable action. The floor ensures it cannot be absolutely suppressed by the scoring
formula; it will always be available to `SelectAction()` as a non-zero candidate.
HOLD's always-generated nature (§3.1.6) means a floor of 0.01 is sufficient to
guarantee the FR-08 fallback requirement.

**Ceiling (1.0):** Utility is a normalised relative score, not an absolute quality
measure. No action should achieve a perfect score — even the best option has some
residual risk or alternative cost. The ceiling prevents the composure noise model
(§3.3) from becoming irrelevant when utility values exceed 1.0.

**Clamp timing:** The clamp is applied once per option, after all four formula components
are multiplied. No intermediate clamping is applied to sub-components.

### 3.2.1.6 Constant Classification and File Rule

**File rule (non-negotiable):** All numeric constants used in utility formulas must be
defined in `UtilityWeights.cs`. No magic numbers may appear in formula code. This rule
is checked in the §9 Approval Checklist as a hard gate.

**Classification tags:**

| Tag | Meaning | Expected proportion |
|-----|---------|---------------------|
| [GT] | Gameplay-tuned: initial estimate set for plausible output; expected to change during gameplay balancing. Not sourced from academic literature. | ~65–70% of DT constants |
| [EST] | Estimated from academic or domain literature. Directionally sourced; specific value not necessarily validated. | ~25–30% |
| [FIXED] | Fixed physical or regulatory constant. Does not change. | ~5–10% |
| [DERIVED] | Mathematically derived from other constants in this specification. No independent tuning. | Rare |

**Governance rules for [GT] constants (enforced by §9 checklist):**
1. Every [GT] constant must have a comment block stating: (a) what it controls,
   (b) the expected effect of increasing it, (c) the safe tuning range.
2. Changing a [GT] constant value in `UtilityWeights.cs` is **not** a spec amendment —
   it is a balance patch. It requires no version bump to this document.
3. Changing the *formula structure* (e.g., adding a new term, changing multiplicative
   to additive) **is** a spec amendment and requires Lead Developer sign-off plus
   a version increment.

## 3.2.2 PASS Utility Formula

### 3.2.2.1 Formula Definition

```
U_PASS(target_i) = BaseUtility_PASS
                 × AttributeMultiplier_PASS
                 × ContextModifier_PASS(target_i)
                 × (1 − RiskPenalty_PASS(target_i))

clamped to [0.01, 1.0]
```

**where:**

```
BaseUtility_PASS     = U_BASE_PASS_NOMINAL × ZoneModifier_PASS(BallZone)
U_BASE_PASS_NOMINAL  = 0.60 [GT]   — pass is the primary positive action; high baseline

AttributeMultiplier_PASS = (0.5 + A_Vision   × 0.5) ^ PASS_VISION_EXP
                         × (0.5 + A_Passing  × 0.5) ^ PASS_TECHNIQUE_EXP

PASS_VISION_EXP      = 0.30 [GT]   — Vision: governs pass lane reading quality
PASS_TECHNIQUE_EXP   = 0.40 [GT]   — Passing: governs execution confidence

ContextModifier_PASS = option.AdjustedPassLaneScore
                     ← populated in §3.1.3.5; range [0.10, 1.00]

RiskPenalty_PASS     = P × (1.0 − A_Passing) × PASS_RISK_COEFF
PASS_RISK_COEFF      = 0.30 [GT]
P                    = DecisionContext.PressureScalar (from PerceptionSnapshot.PressureScalar)
```

**Attribute rationale:**

- Both Vision and Passing use the shifted form `(0.5 + A_x × 0.5)` to prevent near-zero
  AttributeMultiplier for players with low passing attributes. Even a poor passer (rating 1)
  retains 50% of the attribute multiplier — they can still attempt a pass; they will just
  be less accurate (governed by Pass Mechanics #5, not by this score).
- `PASS_VISION_EXP = 0.30` applies a diminishing return curve. Moving from Vision=1 to
  Vision=10 raises the shifted value from 0.5 to ~0.737, contributing `0.737^0.3 = 0.908`.
  Moving from Vision=10 to Vision=20 raises it to 1.0, contributing `1.0^0.3 = 1.000`.
  The difference between a poor and an average Vision player is larger than between
  average and elite — consistent with the principle that minimal competence matters most.
- `PASS_TECHNIQUE_EXP = 0.40` applies a slightly steeper curve. Passing accuracy has
  greater within-option variance than Vision; a high-Passing player's utility should
  rise more as their execution capability improves.

**`ContextModifier_PASS` rationale:**
The pass lane score (`AdjustedPassLaneScore`, defined in §3.1.3.5) already incorporates
the `GoalDirectionModifier` (forward pass bonus), obstacle density, and distance
weighting. It serves directly as the context modifier — a clean pass to a forward
teammate gets close to 1.0; a backward pass through a crowded lane gets ≈ 0.1. No
further context adjustment is needed at this stage; `TacticalModifier_PASS` is
Stage 1+ only (1.0 at Stage 0).

### 3.2.2.2 Sub-Component Derivations

**BaseUtility at zone extremes:**

| BallZone | ZoneModifier | BaseUtility |
|----------|-------------|-------------|
| DEFENSIVE | 1.05 | 0.60 × 1.05 = **0.630** |
| MIDFIELD | 1.00 | 0.60 × 1.00 = **0.600** |
| ATTACKING | 0.90 | 0.60 × 0.90 = **0.540** |

**AttributeMultiplier at attribute extremes (shifted form):**

| Vision | Passing | Shifted_V | Shifted_P | AM = Sv^0.3 × Sp^0.4 |
|--------|---------|-----------|-----------|----------------------|
| 1 | 1 | 0.500 | 0.500 | 0.500^0.3 × 0.500^0.4 = 0.812 × 0.758 = **0.615** |
| 1 | 20 | 0.500 | 1.000 | 0.812 × 1.000 = **0.812** |
| 20 | 1 | 1.000 | 0.500 | 1.000 × 0.758 = **0.758** |
| 20 | 20 | 1.000 | 1.000 | 1.000 × 1.000 = **1.000** |
| 10 | 10 | 0.737 | 0.737 | 0.737^0.3 × 0.737^0.4 = 0.908 × 0.882 = **0.801** |

**RiskPenalty at extremes:**

| P | A_Passing | RiskPenalty = P × (1−AP) × 0.30 |
|---|-----------|--------------------------------|
| 0.0 | any | 0.000 |
| 1.0 | 0.0 (raw=1) | 1.0 × 1.0 × 0.30 = **0.300** |
| 1.0 | 1.0 (raw=20) | 1.0 × 0.0 × 0.30 = **0.000** |

### 3.2.2.4 Dominated-Action Boundary Analysis

**Question: Under what conditions does PASS dominate all other possession options?**

PASS achieves maximum U_PASS ≈ 0.54–0.63 (elite attributes, clean lane, defensive zone).
For PASS to dominate, it must exceed SHOOT, DRIBBLE, and HOLD at the same inputs.

SHOOT in the attacking zone achieves U_SHOOT ≈ 0.7+ for agents with Finishing > 15
and good GoalOpeningScore — SHOOT correctly dominates PASS in the attacking third when
the agent has a clear sight of goal and good finishing ability. PASS dominates in the
defensive and midfield zones where SHOOT is heavily penalised by zone modifier.

DRIBBLE achieves ≈ 0.35–0.45 in open space — PASS dominates DRIBBLE in most cases,
as designed. DRIBBLE should win only when space is available and the agent has high
dribbling attributes while passing options are poor (congested lanes).

HOLD at ≈ 0.25 baseline — PASS correctly dominates HOLD in all normal conditions.
HOLD only wins when all pass lanes are blocked (ContextModifier ≈ 0.10) and pressure
is high, pulling PASS below 0.064 × ZoneModifier. This is exactly the intended HOLD
fallback behaviour.

**No pathological dominance identified.** PASS dominates correctly in midfield and
defence; SHOOT correctly takes over in the attacking third for capable finishers.
